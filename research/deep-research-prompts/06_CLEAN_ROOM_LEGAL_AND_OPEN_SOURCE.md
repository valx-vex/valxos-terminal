# Master Prompt: Clean Room, Legal, And Open Source Strategy Research

You are researching governance and implementation boundaries for `VALXOS Terminal`.

Context:
- We want strong compatibility with existing ecosystems
- We also want to stay on the right side of clean-room engineering and public/open distribution
- This is research, not legal advice

Research questions:
- What are the best practices for clean-room compatible implementations?
- How do open-source projects usually document interface compatibility without copying protected implementation?
- What risks arise when studying public APIs versus leaked/internal materials?
- What boundaries should the team set around source analysis, documentation, tests, naming, UI imitation, and code reuse?
- What open-source licenses or project structures are best suited to a free/public developer tool with plugins and optional hosted services?
- How should attribution, notices, dependency hygiene, and contribution rules be handled?

Deliverables:
- Clean-room operating guidelines
- Risk map
- Red lines and safe zones
- Recommended repo governance rules
- Suggested licensing and contribution approach
- Practical checklist for researchers and implementers

Output format:
- Markdown
- Clearly label `Not Legal Advice`
- Separate `Observed Practice` from `Suggested Policy`
- Include primary sources and authoritative references where possible
